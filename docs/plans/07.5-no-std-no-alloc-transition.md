# no_std / no_alloc 전환 및 Host Simulation 유지 Plan

> 목적: FlashDB-for-rust를 임베디드 중심 라이브러리로 재정렬하기 위해 core를 `no_std` + `no_alloc` 기준으로 전환하되, Linux/host 환경에서의 빠른 회귀 테스트와 file-backed simulation 능력은 유지하는 설계/구현 계획을 정의한다.

## 1. 목표

이 단계의 핵심 목표는 두 가지를 동시에 만족하는 것이다.

1. core library가 heap allocator 없이도 동작해야 한다.
2. Linux 상에서 mock/file 기반 테스트를 계속 빠르게 돌릴 수 있어야 한다.

즉, 방향은 다음과 같다.
- "embedded target에서 안정적으로 돌아가는 구현"을 기준으로 core를 다시 설계하고,
- "host에서 실행 가능한 포팅 레이어/테스트 하네스"를 별도로 유지한다.

이 문서는 plan 07의 세부화 문서이며, 특히 아래 질문에 대한 구현 방침을 고정한다.
- `Vec` 같은 alloc 자료구조를 제거해도 Linux에서 테스트 가능한가?
- 가능하다면 어떤 계층 분리로 가져가야 하는가?

대답은 "가능하다"이며, 핵심은 core와 host porting layer를 분리하는 것이다.

## 2. 범위

### 포함
- crate 전반의 `alloc` 의존 지점 식별
- `Vec`/heap 기반 runtime state를 fixed-capacity / caller-provided buffer 기반으로 바꾸는 설계
- no_std + no_alloc 기준의 public/internal API 재설계
- host-side test strategy 재정의
- file-backed simulation을 core 밖의 std-only support layer로 유지하는 방향 정리
- embedded smoke example이 allocator 없이 빌드/동작 가능하도록 재설계하는 방향 정의

### 포함하지 않음
- 이 문서 자체에서 전체 no_alloc 전환 구현 완료
- 실제 flash hardware backend 신규 구현
- property/fuzz test 도입 자체의 완료

## 3. 핵심 원칙

### 3.1 core는 no_std + no_alloc
core library (`src/`)는 다음을 목표로 한다.
- `extern crate alloc` 제거
- heap allocator 없이 동작
- `Vec`, `String`, `Box`, `Arc`, `Rc` 등 동적 할당 타입 미사용

### 3.2 host 테스트 가능성은 유지
no_alloc 전환은 Linux 테스트 포기를 의미하지 않는다.
오히려 구조는 아래처럼 더 명확해져야 한다.

- core: no_std + no_alloc
- host test backend:
  - RAM array 기반 mock flash
  - 필요 시 std feature/file-backed simulator
- test harness:
  - Linux에서 `cargo test`로 실행

즉, "core는 allocator 없이"와 "host 테스트 가능"은 충돌하지 않는다.

### 3.3 alloc 제거는 API 설계 문제다
동적 할당을 제거하려면 단순히 `Vec`를 배열로 바꾸는 것이 아니라 API 경계도 바꿔야 한다.
예를 들면:
- owned iterator snapshot -> streaming iterator/state machine
- owned payload return -> caller-provided output buffer
- runtime-discovered variable list -> fixed-capacity table 또는 on-demand scan

### 3.4 std 전용 지원 코드는 분리
file-backed simulation, 긴 시나리오 테스트, process restart 기반 reboot 검증은 여전히 유용하다.
다만 이들은 core에 섞지 않고 별도 std-only layer로 유지한다.

## 6. 세부 구현 단계

### Phase 1. alloc 의존 지점 전수조사

목표:
- 현재 crate에서 heap allocation이 실제로 어디서 필요한지 전부 식별한다.

반드시 분류할 것:
- `Vec` 기반 runtime metadata
- iterator snapshot 수집
- payload 임시 버퍼
- test-only allocation
- std-only support code

산출물:
- 파일별 alloc 사용 목록
- 각 사용처에 대한 대체 전략 메모

우선순위 분류:
1. public API에 드러나는 alloc
2. runtime hot path에 있는 alloc
3. mount/recovery에서만 쓰는 alloc
4. test-only alloc

### Phase 2. public API를 no_alloc 친화적으로 재정의

목표:
- core API가 owned collection 반환에 기대지 않도록 바꾼다.

권장 방향:
- `iter()`가 내부 snapshot `Vec`를 만들지 않게 한다.
- range query도 가능하면 streaming/filter iterator로 바꾼다.
- payload read는 항상 caller-provided buffer 기반으로 정리한다.

예시 방향:
- `iter() -> Iterator<Item = OwnedRecord>`
  대신
- `iter() -> KvIterator<'_, F>` 같은 상태 기반 iterator
- `collect snapshot` 대신 `next()` 때 flash를 읽는 방식

주의:
- iterator가 borrow/lifetime 때문에 너무 복잡해지면,
  먼저 callback 기반 API를 도입한 뒤 iterator sugar를 나중에 얹는 것도 허용한다.

### Phase 3. KVDB alloc 제거

목표:
- KVDB의 mount / scan / iterator / GC 흐름에서 heap 의존을 제거한다.

집중 항목:
- live-record snapshot 제거
- GC live set 수집 방식 변경
- integrity/report 자료구조가 alloc에 의존하면 고정 용량 또는 callback 기반으로 전환

권장 접근:
- sector/record traversal은 on-demand scan 우선
- host에서 느려지는 것은 허용하되 correctness를 먼저 유지
- 이후 필요 시 fixed-capacity cache를 도입

### Phase 4. TSDB alloc 제거

목표:
- TSDB의 sector table, iter snapshot, range filtering snapshot을 heap 없이 동작하게 만든다.

핵심 문제:
- 현재 TSDB는 sector runtime table과 record snapshot에 동적 메모리를 쓰기 쉬운 구조다.

대체 방향:
- sector runtime metadata는
  - compile-time bounded table 또는
  - caller-provided scratch buffer 또는
  - mount-time full cache 없이 on-demand scan
  중 하나를 택해야 한다.

권장 순서:
1. snapshot 기반 iter 제거
2. sector runtime state의 저장 방식을 고정 용량화
3. range/count를 full snapshot 없이 구현

중요:
- no_alloc 전환 초기에 성능이 다소 나빠져도 괜찮다.
- correctness와 bounded memory가 우선이다.

### Phase 5. scratch buffer 전략 정리

목표:
- 코어에서 필요한 임시 버퍼를 heap 대신 어떤 방식으로 받을지 통일한다.

선택지:
- stack의 소형 고정 배열
- 호출자 제공 scratch buffer
- config 기반 고정 최대치
- `heapless` 같은 bounded container

권장 원칙:
- payload 전체를 항상 메모리에 담지 말고, 가능한 한 chunked read를 우선한다.
- write path도 write granularity 단위 scratch만 필요하도록 유지한다.

### Phase 6. host-side simulation 유지 전략

목표:
- Linux에서 여전히 빠르게 테스트할 수 있도록 std-only 포팅 레이어를 정리한다.

핵심 방침:
- core가 no_alloc/no_std가 되어도 host 테스트는 그대로 가능하다.
- `MockFlash`는 계속 핵심 테스트 수단으로 유지한다.
- file-backed reboot/crash simulation은 std feature 또는 tests support 모듈로 둔다.

권장 구조:
- core: std 비의존
- `storage/file_sim.rs`: `#[cfg(feature = "std")]` 또는 tests-only support
- 긴 reboot simulation은 host에서 subprocess/file 기반으로 계속 실행

즉, 원본 FlashDB의 Linux mode와 같은 역할은 유지하되, core에 std를 섞지 않는다.

### Phase 7. embedded example을 no_alloc smoke로 재설계

목표:
- 현재 allocator를 쓰는 Embassy example을 최종적으로 allocator 없는 smoke example으로 바꾼다.

권장 방향:
- library core가 no_alloc이 되면 example에서도 heap 초기화를 제거
- `MockFlash<[u8; N]>` 또는 실제 backend를 그대로 사용하되,
  example 내부에서 alloc crate를 전혀 초기화하지 않는 상태로 build 성공 확인

완료 기준:
- `examples/stm32f401re/src/bin/flashdb.rs`가 allocator 없이 빌드
- `examples/nrf5340/src/bin/flashdb.rs`가 allocator 없이 빌드

### Phase 8. 검증 루틴 재정의

목표:
- no_alloc 전환 이후의 검증 절차를 문서와 스크립트에 반영한다.

반드시 유지할 검증:
- `cargo test`
- `cargo test --features std`
- embedded example build

추가 권장 검증:
- core library 자체를 embedded target에 대해 `cargo check --lib --target ...`
- `alloc` feature가 비활성화된 상태 확인
- 필요 시 `rg "alloc::|Vec<|String|Box<" src/` 같은 점검 스크립트

## 7. Linux에서의 테스트 가능성에 대한 결론

질문:
- no_std + no_alloc로 가면 Linux에서 돌려서 테스트할 수 있는가?

답:
- 가능하다.

전제:
- 테스트 대상은 core logic이고,
- Linux는 실제 flash 하드웨어 대신 mock/file 기반 porting layer를 제공하면 된다.

즉, 다음 구조를 취하면 된다.
- core logic: no_std + no_alloc
- Linux test harness: std 사용 가능
- Linux flash port: RAM/file 기반 시뮬레이터

이 방식은 원본 FlashDB의 file mode / porting layer와 같은 철학이다.
차이는 Rust 쪽에서 core의 allocator 의존을 더 엄격하게 제거한다는 점이다.

## 8. 방법론

### 8.1 alloc 제거는 작은 단계로 진행
한 번에 전체를 바꾸지 않는다.
- 먼저 alloc 사용처 목록화
- 그 다음 API 경계 수정
- 이후 KV/TSDB 내부 구현 수정
- 마지막으로 examples와 검증 스크립트 정리

### 8.2 correctness를 먼저 유지
성능 최적화보다 우선할 것:
- bounded memory
- deterministic behavior
- reboot/recovery semantics 유지

### 8.3 host test는 포기하지 않는다
no_alloc 전환 중에도 Linux 테스트는 계속 살아 있어야 한다.
그래야 iteration 속도가 유지된다.

### 8.4 regression test를 함께 옮긴다
alloc 제거 과정에서 발견된 모든 버그는 regression test로 묶는다.
