# FlashDB-for-embassy 진행 현황

작성일: 2026-04-18
이 문서는 "지금 어디까지 구현됐는지"와 "다음에 무엇을 해야 하는지"를 progress 파일 하나만 보고 바로 파악할 수 있도록 유지한다.

## 1. 현재 기준점

- 현재 주력 plan: `docs/plans/02-storage-layout-and-status-foundation.md`
- 전체 진행 위치:
  - plan 00: 해석 완료
  - plan 01: crate/module 구조 작업 완료
  - plan 02: 일부 핵심 foundation 구현 완료, 하지만 아직 미완료
  - plan 03 이후: 아직 본격 시작 전

즉, 지금 프로젝트 상태는:
- "구조만 있는 초기 스캐폴드" 단계는 지났고,
- "storage/status/aligned-write 같은 공통 기반"을 깔아 둔 상태이며,
- 다음 작업은 plan 02의 남은 codec 정밀화 또는 plan 03(blob layer)로 넘어가기 전 필요한 foundation 마무리다.

### 2.3 plan 02에서 이번까지 구현된 것

#### storage region / offset foundation
- `src/storage/region.rs`
  - region validation
  - sector count 계산
  - logical offset 검사
  - absolute address 변환
  - sector 시작 주소 계산
  - sector index 계산

#### storage adapter foundation
- `src/storage/nor_flash.rs`
  - `NorFlashRegion::new(...)`에서 backend geometry와 region geometry 일치 여부 검증
  - region-relative read/write 지원
  - `write_aligned(...)` 지원
  - unaligned offset 거부
  - tail write를 erased byte(0xFF)로 padding

#### mock NOR flash foundation
- `src/storage/mock.rs`
  - RAM 기반 mock flash
  - NOR의 1 -> 0만 허용하는 semantics 검증 가능

#### alignment / CRC / common constants foundation
- `src/layout/align.rs`
- `src/crc.rs`
- `src/layout/common.rs`

구현된 핵심 내용:
- align helper
- CRC32 helper
- upstream FlashDB 기준 magic/sentinel 정리
- `DATA_UNUSED_SENTINEL` 주의 주석 추가

#### status table foundation
- `src/layout/status.rs`

구현된 핵심 내용:
- 1-bit granularity 지원
- whole-byte granularity 지원
- status table 길이 계산
- encode/decode
- `encode_transition()`
- `transition_write_span()`
- `StatusTableBuf`

현재 status 쪽은 "단순 placeholder"가 아니라 실제 foundation 역할을 하는 수준까지 올라와 있다.

## 5. 아직 안 끝난 것

중요: plan 02는 아직 완료가 아니다.

### 5.1 plan 02에서 남은 핵심 작업
1. `src/layout/kv.rs` 정밀화
   - KV sector header를 upstream FlashDB layout에 더 가깝게 맞추기
   - KV record header field/length/padding/CRC 범위 계산 정밀화
   - malformed header 처리 강화

2. `src/layout/ts.rs` 정밀화
   - TS sector header 구조를 upstream에 더 가깝게 맞추기
   - log index 구조와 fixed/variable blob mode 계산 정밀화
   - sector remain / index-data dual-ended layout helper 보강

3. status-table의 flash I/O helper 추가 여부 결정 및 구현
   - upstream `_fdb_write_status`, `_fdb_read_status`에 대응하는 helper가 아직 없음
   - 지금은 codec + transition span까지만 있음
   - 실제 backend write/read와 결합할 공통 helper가 필요함

### 5.2 아직 스캐폴드 또는 초기 수준인 영역
- `src/storage/file_sim.rs`
  - std-gated scaffold 수준
- `src/kv/*`
  - 구조는 있으나 실제 정책/동작은 미구현에 가까움
- `src/tsdb/*`
  - 구조는 있으나 실제 정책/동작은 미구현에 가까움

## 6. 다음 작업 우선순위

후속 작업자는 아래 순서를 추천한다.

### 추천 순서 A: plan 02를 먼저 마무리
가장 안전한 다음 액션:
1. `docs/plans/02-storage-layout-and-status-foundation.md`를 다시 기준 문서로 잡기
2. `src/layout/kv.rs` 정밀화
3. `src/layout/ts.rs` 정밀화
4. status flash I/O helper 추가
5. 관련 테스트 확장

이 경로가 좋은 이유:
- plan 03(blob), plan 04(KV MVP), plan 06(TSDB) 모두 layout 정확도에 크게 의존함
- foundation이 덜 끝난 상태에서 상위 로직으로 가면 나중에 구조를 다시 뜯어야 할 가능성이 큼

### 추천 순서 B: plan 03으로 넘어가기
조건부로 가능:
- plan 02의 남은 gap을 "지금 당장 blob layer 구현에 치명적이지 않은 수준"이라고 판단할 때
- 이 경우에도 최소한 KV/TS header codec의 큰 틀은 다시 확인하고 넘어가는 것이 좋음
