## 2. 이번에 plan 03에서 완료한 것

### 2.1 BlobRef / BlobBuf 역할 분리
`src/blob/reader.rs`에서 blob borrowed-slice API를 재정리했다.

구현된 내용:
- `BlobRef<'a>`
  - 쓰기 입력용 borrowed payload wrapper
  - `len`, `is_empty`, `as_bytes` 제공
- `BlobBuf<'a>`
  - 읽기 대상 caller-provided mutable buffer wrapper
  - `len`, `is_empty`, `as_bytes`, `as_mut_bytes` 제공

의미:
- 기존처럼 locator와 payload buffer를 한 타입에 섞지 않고,
- raw payload wrapper와 on-flash 위치 정보 타입을 분리했다.

### 2.2 BlobLocator / KV / TS locator 계층 정립
`src/blob/locator.rs`를 plan 03 요구 수준으로 확장했다.

구현된 내용:
- `BlobLocator`
  - `meta_offset`
  - `data_offset`
  - `len`
  - `end_offset()` helper
- 생성 시 `StorageRegion` 경계 검증
  - metadata offset 범위 검증
  - payload range 범위 검증
- 의미 분리용 wrapper
  - `KvValueLocator`
  - `TsPayloadLocator`

의미:
- upstream `fdb_blob.saved.meta_addr / addr / len`를 Rust 타입으로 분리 캡슐화했고,
- KV payload와 TS payload의 의미 차이를 타입 레벨에서 드러낼 수 있게 됐다.

### 2.3 exact / truncated / chunk read semantics 구현
`src/blob/reader.rs`에 공통 blob reader 계층을 추가했다.

구현된 내용:
- `BlobStorage` trait
  - region-relative read backend 추상화
- `BlobReader<S>`
  - `blob_len(locator)`
  - `read_exact(locator, &mut [u8])`
  - `read_truncated(locator, &mut [u8])`
  - `read_chunk(locator, offset, &mut [u8])`
- `BlobCursor`
  - chunked sequential read helper
- `NorFlashRegion<F>`에 대해 `BlobStorage` 구현

에러/정책:
- exact read에서 buffer가 너무 작으면 `Error::BufferTooSmall`
- chunk offset이 payload len을 넘으면 `Error::InvalidBlobOffset`
- offset == len 이면 0-byte read 허용

의미:
- 원본 `fdb_blob_read`의 truncation semantics를 유지하면서,
- exact / partial / sequential chunk read를 명시적 API로 분리했다.

### 2.4 codec layer를 raw byte layer와 분리
`src/blob/codec.rs`를 재설계했다.

구현된 내용:
- `EncodeToBytes`
- `DecodeFromBytes`
- 둘을 묶는 marker trait `BlobCodec`

원칙 반영:
- DB core는 raw bytes만 안다.
- typed payload 해석/직렬화는 codec 계층이 담당한다.
- transmute-style direct struct serialization을 기본 API로 강제하지 않는다.

### 2.5 plan 03 검증 테스트 추가
테스트를 blob 계층 중심으로 보강했다.

추가/수정된 테스트:
- `tests/blob_layer.rs`
  - BlobRef / BlobBuf semantics
  - locator validation
  - truncated/exact/chunk/cursor read
  - buffer-too-small / invalid-offset 오류 케이스
- `tests/blob_codec.rs`
  - typed payload encode/decode roundtrip
  - encode/decode contract error 검증

## 5. plan 03 완료 판단

이번 기준에서 plan 03 완료로 판단한 이유:
- KV/TS가 공통으로 사용할 locator 타입이 생겼다.
- caller가 payload 전체 길이를 조회할 수 있다.
- exact / truncated / partial read를 선택적으로 사용할 수 있다.
- raw bytes 계층과 typed codec 계층이 분리되었다.
- no_std 친화적인 borrowed-slice 중심 API를 유지했다.

보수적으로 보면 TSDB의 fixed/variable payload locator 계산은 실제 append/query 정책이 들어가는 plan 06에서 소비될 부분이 남아 있다.
하지만 plan 03의 공통 blob/codec 계층 완료 기준 자체는 이번 slice로 충족했다.

## 6. 아직 스캐폴드 또는 다음 plan 대상인 영역

이건 plan 03 미완료가 아니라 상위 plan의 미구현 영역이다.

- `src/kv/*`
  - 실제 mount / scan / set / delete / update 정책은 plan 04~05 대상
- `src/tsdb/*`
  - 실제 append / query / recovery / fixed-vs-variable payload placement 정책은 plan 06 대상
- codec utility 상위 helper
  - `set_encoded`, `get_decoded` 같은 DB-facing convenience API는 실제 KV/TS read/write surface가 생기는 단계에서 붙이는 것이 적절함
